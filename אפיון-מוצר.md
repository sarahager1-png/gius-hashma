## 1. חזון המוצר

### הבעיה
רשת אהלי יוסף יצחק מנהלת עשרות מוסדות חינוך הזקוקים מדי שנה למורות ומחנכות. תהליך הגיוס התנהל ידנית — גיליונות Excel, שיחות טלפון, ניירות — ללא יכולת מעקב, סינון, או שקיפות לכל הצדדים.

### הפתרון
פלטפורמה דיגיטלית מרכזית המאחדת את כל שרשרת הגיוס:
- **מועמדות** מגישות מועמדות מאיפה שהן
- **מוסדות** מפרסמים משרות ומנהלים הגשות
- **הנהלת הרשת** רואה תמונה מלאה + אנליטיקה

### יעדים עסקיים
1. קיצור זמן גיוס ממוצע ב-60%
2. הגברת שקיפות התהליך לכל הצדדים
3. ניהול מרכזי של כל פעילות הגיוס ברשת
4. נגישות ניידת לכל המשתמשים

## 3. ארכיטקטורה ו-Stack

### Stack טכנולוגי

| שכבה | טכנולוגיה | גרסה | הערה |
|------|-----------|-------|------|
| Framework | Next.js App Router | 16.2.4 | SSR + Client islands |
| Runtime | React | 19.2.4 | |
| Language | TypeScript | 5.x | strict mode |
| Styling | Tailwind CSS | v4 | RTL enabled |
| UI Components | shadcn/ui + Radix UI | — | |
| Charts | Recharts | — | אנליטיקה |
| State | Zustand | — | global state |
| Data Fetching | TanStack React Query | v5 | caching + mutations |
| Toasts | Sonner | — | הודעות משתמש |
| Export | XLSX | — | ייצוא לאקסל |
| Database | Supabase (PostgreSQL) | — | |
| Auth | Supabase Auth | — | email/password |
| Deploy | Vercel | — | auto-deploy |

### עקרונות ארכיטקטוניים
- **Server Components** — כל הדפים שולפים נתונים server-side
- **Client Islands** — רכיבים אינטראקטיביים (טפסים, פילטרים, טבלאות) כ-client components
- **Two Supabase clients:**
  - `createClient()` — SSR, מכבד RLS
  - `createServiceClient()` — Service Role, עוקף RLS (לפעולות מנהל בלבד)
- **Middleware** — מגן על routes, מנהל sessions

## 4. מסד הנתונים

> ملف schema מלא: `supabase/schema.sql`

### טבלת `profiles` — בסיס כל המשתמשים
```sql
id            UUID PRIMARY KEY  -- = auth.users.id
role          TEXT NOT NULL     -- 'מועמדת' | 'מוסד' | 'מנהל רשת' | 'אדמין מערכת'
full_name     TEXT
phone         TEXT
created_at    TIMESTAMPTZ DEFAULT now()
```

**RLS:**
- משתמש רואה רשומה שלו בלבד
- אדמין רואה הכל
- Service Role יכול ליצור רשומות

---

### טבלת `candidates` — פרופיל מועמדת
```sql
id                  UUID PRIMARY KEY DEFAULT gen_random_uuid()
profile_id          UUID REFERENCES profiles(id)
city                TEXT
college             TEXT
graduation_year     INTEGER
specialization      TEXT   -- 'גן ילדים' | 'יסודי' | 'שניהם'
academic_level      TEXT   -- 'סטאג'' | 'בוגרת' | 'מנוסה'
years_experience    INTEGER DEFAULT 0
availability_status TEXT   -- ראה ערכים למטה
placement_location  TEXT
supervisor_name     TEXT
supervisor_phone    TEXT
birth_year          INTEGER
prev_employer       TEXT
prev_role           TEXT
special_skills      TEXT
marital_status      TEXT   -- 'רווקה' | 'נשואה'
has_cv              BOOLEAN DEFAULT false
cv_url              TEXT
bio                 TEXT
created_at          TIMESTAMPTZ DEFAULT now()
updated_at          TIMESTAMPTZ DEFAULT now()  -- מעודכן ע"י trigger
```

**ערכי סטטוס זמינות:**
| ערך | משמעות |
|-----|---------|
| `מחפשת סטאג'` | סטודנטית הזקוקה למקום סטאג' |
| `בוגרת מחפשת משרה` | בוגרת מחפשת תפקיד מלא |
| `פתוחה להצעות` | לא מחפשת אקטיבית אך תשקול |
| `משובצת` | כבר בתפקיד, לא זמינה |
| `לא פעילה` | לא רלוונטית כרגע |

**RLS:**
- מועמדת עורכת פרופיל שלה בלבד
- מוסדות יכולים לקרוא את כל המועמדות
- אדמין יכול לעדכן הכל

---

### טבלת `institutions` — מוסד חינוכי
```sql
id                UUID PRIMARY KEY DEFAULT gen_random_uuid()
profile_id        UUID REFERENCES profiles(id)
institution_name  TEXT NOT NULL
city              TEXT
address           TEXT
phone             TEXT
institution_type  TEXT   -- 'גן ילדים' | 'בית ספר יסודי' | 'מוסד אחר'
is_approved       BOOLEAN DEFAULT false
approved_at       TIMESTAMPTZ
approved_by       UUID REFERENCES profiles(id)
created_at        TIMESTAMPTZ DEFAULT now()
```

**RLS:**
- מוסד עורך פרופיל שלו
- מועמדות רואות מוסדות **מאושרים** בלבד
- אדמין מנהל הכל

---

### טבלת `jobs` — משרות פתוחות
```sql
id              UUID PRIMARY KEY DEFAULT gen_random_uuid()
institution_id  UUID REFERENCES institutions(id)
title           TEXT NOT NULL
description     TEXT
city            TEXT
specialization  TEXT   -- 'גן ילדים' | 'יסודי' | 'שניהם'
job_type        TEXT   -- 'סטאג'' | 'חלקי' | 'מלא'
status          TEXT   DEFAULT 'פעילה'
start_date      DATE
expires_at      DATE
created_at      TIMESTAMPTZ DEFAULT now()
updated_at      TIMESTAMPTZ DEFAULT now()
```

**מחזור חיי משרה:**
```
פעילה ──→ מושהית ──→ פעילה  (הפשרה)
פעילה ──→ אוישה           (שובצה מועמדת)
פעילה ──→ בוטלה           (ביטול ידני)
פעילה ──→ פג תוקפה        (expires_at עבר)
```

**RLS:**
- מוסד מנהל משרות שלו בלבד
- מועמדות רואות משרות `פעילה` של מוסדות מאושרים
- אדמין רואה הכל

---

### טבלת `applications` — הגשות מועמדות
```sql
id                  UUID PRIMARY KEY DEFAULT gen_random_uuid()
job_id              UUID REFERENCES jobs(id)
candidate_id        UUID REFERENCES candidates(id)
status              TEXT DEFAULT 'ממתינה'
cover_letter        TEXT
institution_notes   TEXT
applied_at          TIMESTAMPTZ DEFAULT now()
updated_at          TIMESTAMPTZ DEFAULT now()
UNIQUE(job_id, candidate_id)  -- מניעת הגשה כפולה
```

**מחזור חיי הגשה:**
```
ממתינה ──→ נצפתה ──→ התקבלה ✓
                └──→ נדחתה  ✗
ממתינה ──→ בוטלה   (המועמדת מבטלת)
```

**RLS:**
- מועמדת מנהלת הגשות שלה
- מוסד רואה ומעדכן הגשות למשרותיו
- אדמין רואה הכל

---

CREATE TRIGGER update_candidates_updated_at

## 7. ממשקי API

### הגשות

| Method | URL | תיאור | גישה |
|--------|-----|--------|------|
| `POST` | `/api/applications` | הגשה חדשה + בדיקת כפילות (409) | מועמדת |
| `PATCH` | `/api/applications/[id]` | עדכון סטטוס / הערות מוסד | מוסד / אדמין |

```typescript
// POST body
{ jobId: string, coverLetter?: string }

// PATCH body
{ status?: string, institution_notes?: string }
```

| Method | URL | תיאור | גישה |

| `PATCH` | `/api/jobs/[id]` | עדכון משרה | מוסד / אדמין |

```typescript
// שדות מותרים לעדכון:
title, description, city, specialization, job_type, status, expires_at
```

## 11. אבטחה ו-RLS

### שכבות הגנה

**שכבה 1 — Middleware (Next.js)**
- מגן על כל routes
- מרענן session cookies
- מפנה לא-מחובר ל-/login

**שכבה 2 — API Routes (Server-side)**
- `supabase.auth.getUser()` בכל endpoint
- בדיקת `role` לפני כל פעולה
- Field whitelist ב-PATCH (מניעת column injection)
- Allowed status values (מניעת ערכים לא-חוקיים)

**שכבה 3 — Supabase RLS**
- מדיניות גישה לפי row ב-PostgreSQL
- גם אם ה-API נפרץ — DB מגן על עצמו

### אימות בעיות אבטחה שטופלו
| בעיה | תיקון |
|------|-------|
| כל מוסד יכול לעדכן משרות של מוסד אחר | Ownership check בAPI + RLS |
| מועמדת יכולה לשנות כל שדה בפרופיל | Whitelist מפורש של שדות מותרים |
| גישה לדוחות ללא role check | role check בכל API של אדמין |
| הגשה כפולה | UNIQUE constraint ב-DB |

### Environment Variables נדרשים
```bash
NEXT_PUBLIC_SUPABASE_URL=https://[project].supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=[JWT anon key]
SUPABASE_SERVICE_ROLE_KEY=[JWT service role key]
NEXT_PUBLIC_APP_URL=https://giuus.vercel.app
```

## 12. מצב דמו ובדיקות

### Bypass Mode

**הפעלה:** cookie `g_pass=giuus2025`  
**כניסה:** `/enter` → מגדיר cookie → מפנה ל-/dashboard

**מה מאפשר:**
- גישה לכל הממשק ללא משתמש אמיתי
- מתג תפקידים בממשק (כניסה לתצוגת מועמדת / מוסד / מנהל)
- נתונים מ-`lib/bypass-data.ts` (hardcoded)
- API routes מחזירים `{ ok: true }` ללא כתיבה ל-DB

**מה לא עושה:**
- לא כותב לבסיס הנתונים
- לא שולח התראות
- לא מפעיל לוגיקה אמיתית

**נתוני דמו כוללים:**
- מועמדות לדוגמה עם פרופילים מלאים
- מוסדות: חלק מאושרים, חלק ממתינים
- משרות פתוחות מכמה מוסדות
- הגשות בסטטוסים שונים
