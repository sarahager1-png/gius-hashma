## 1. סקירה כללית

### מהי המערכת?
פלטפורמת גיוס והשמה דיגיטלית לרשת בתי הספר אהלי יוסף יצחק (חב"ד). המערכת מחברת בין מועמדות (סטודנטיות להוראה ובוגרות) לבין מוסדות חינוך ברשת — בצורה חכמה, אוטומטית ומהירה.

### בעיה שהמערכת פותרת
בעבר, תהליך הגיוס נעשה ידנית — טפסי Excel, שיחות טלפון, ניהול מידע מפוזר. המערכת ממרכזת את כל התהליך: רישום, חיפוש, התאמה, הגשת מועמדות, תקשורת ומעקב — במקום אחד.

### משתמשים עיקריים
| סוג | מספר משוער | שימוש |
|-----|------------|--------|
| מועמדות | עשרות–מאות | חיפוש עבודה, הגשת מועמדויות |
| מוסדות (גנים, בתי-ספר) | ~20–50 | פרסום משרות, ניהול הגשות |
| מנהלי רשת | 2–5 | פיקוח, התאמה, אישורים |

## 2. משתמשים וסמכויות

### תפקידים במערכת

#### מועמדת (`מועמדת`)
- הרשמה ועריכת פרופיל אישי
- העלאת קורות חיים (PDF/Word)
- חיפוש וסינון משרות פתוחות
- הגשת מועמדות עם מכתב כוונות
- מעקב סטטוס הגשות (ציר זמן ויזואלי)
- קבלת הצעות התאמה מבוססות AI
- קבלת עדכונים ב-WhatsApp

#### מוסד חינוכי (`מוסד`)
- הרשמה (דורשת אישור מנהל רשת)
- עריכת פרופיל המוסד
- פרסום משרות פתוחות
- צפייה בהגשות שהתקבלו
- עדכון סטטוס הגשות (נצפתה / התקבלה / נדחתה)
- קבלת המלצות מועמדות מבוססות AI
- קביעת ראיונות
- קבלת התראות WhatsApp על הגשות חדשות

#### מנהל רשת (`מנהל רשת`)
- כל סמכויות המוסד
- אישור רישום מוסדות חדשים
- יצירת מועמדות ידנית במערכת
- צפייה בכל המועמדות והמוסדות
- הפעלת מנוע ההתאמה (AI)
- יצירת הפניות ישירות (referrals)
- לוח בקרה עם KPIs ואנליטיקה
- ניהול הגדרות מערכת (WhatsApp, שנת לימודים)
- ייצוא נתונים

#### אדמין מערכת (`אדמין מערכת`)
- כל סמכויות מנהל רשת
- ניהול משתמשים
- הגדרות WhatsApp ובדיקה
- גישה לדוחות מתקדמים

## 4. מסד הנתונים

### טבלת `profiles` — בסיס כל המשתמשים
```sql
id            UUID PRIMARY KEY  (= Supabase auth.users.id)
role          TEXT  -- 'מועמדת' | 'מוסד' | 'מנהל רשת' | 'אדמין מערכת'
full_name     TEXT
phone         TEXT
created_at    TIMESTAMPTZ
```

### טבלת `candidates` — פרופיל מועמדת
```sql
id                  UUID PRIMARY KEY
profile_id          UUID FK → profiles
city                TEXT
college             TEXT
graduation_year     INTEGER
specialization      TEXT  -- 'יסודי' | 'חינוך מיוחד' | 'יעוץ' | 'גיל הרך' | 'חטיבה' | 'תיכון' | 'אחר'
academic_level      TEXT  -- "שנה ב'-סטאג'" | 'תואר ראשון' | 'תואר שני' | ...
years_experience    INTEGER
availability_status TEXT  -- ראה סטטוסים למטה
placement_location  TEXT
supervisor_name     TEXT
supervisor_phone    TEXT
birth_year          INTEGER
prev_employer       TEXT
prev_role           TEXT
special_skills      TEXT
marital_status      TEXT  -- 'רווקה' | 'נשואה'
has_cv              BOOLEAN
cv_url              TEXT
bio                 TEXT
created_at          TIMESTAMPTZ
updated_at          TIMESTAMPTZ
```

**סטטוסי זמינות מועמדת:**
- `מחפשת סטאג'` — סטודנטית המחפשת מקום סטאג'
- `בוגרת מחפשת משרה` — בוגרת לתפקיד מלא
- `פתוחה להצעות` — לא מחפשת אקטיבית אך תשקול
- `משובצת` — כבר מועסקת
- `לא פעילה` — לא רלוונטית כרגע

### טבלת `institutions` — פרופיל מוסד
```sql
id                UUID PRIMARY KEY
profile_id        UUID FK → profiles
institution_name  TEXT
city              TEXT
address           TEXT
phone             TEXT
institution_type  TEXT  -- גן ילדים / בית ספר יסודי / חטיבה / תיכון / אחר
is_approved       BOOLEAN DEFAULT false
approved_at       TIMESTAMPTZ
approved_by       UUID FK → profiles
created_at        TIMESTAMPTZ
```

### טבלת `jobs` — משרות פתוחות
```sql
id              UUID PRIMARY KEY
institution_id  UUID FK → institutions
title           TEXT
description     TEXT
city            TEXT
specialization  TEXT
job_type        TEXT  -- "סטאג'" | 'חלקי' | 'מלא'
placement_type  TEXT  -- 'מילוי מקום לחופשת לידה' | 'שיבוץ לשנה' | 'שיבוץ קבוע'
start_date      DATE
status          TEXT  -- ראה סטטוסים למטה
expires_at      DATE
created_at      TIMESTAMPTZ
updated_at      TIMESTAMPTZ
```

**סטטוסי משרה:**
- `פעילה` — פרסום פעיל, מקבלת הגשות
- `מושהית` — פרסום מושהה זמנית
- `אוישה` — המשרה אוישה
- `בוטלה` — המשרה בוטלה
- `פג תוקפה` — תאריך תפוגה עבר

### טבלת `applications` — הגשות מועמדות
```sql
id                  UUID PRIMARY KEY
job_id              UUID FK → jobs
candidate_id        UUID FK → candidates
status              TEXT  -- ראה סטטוסים למטה
cover_letter        TEXT
institution_notes   TEXT  -- הערות פנימיות של המוסד
applied_at          TIMESTAMPTZ
updated_at          TIMESTAMPTZ
```

**מחזור חיי הגשה:**
```
ממתינה ──→ נצפתה ──→ התקבלה
                 └──→ נדחתה
         └──→ בוטלה  (המועמדת מבטלת)
```

### טבלת `referrals` — הפניות ישירות
```sql
id              UUID PRIMARY KEY
job_id          UUID FK → jobs
candidate_id    UUID FK → candidates
status          TEXT  -- ראה סטטוסים למטה
interview_date  DATE
notes           TEXT
created_at      TIMESTAMPTZ
updated_at      TIMESTAMPTZ
```

**מחזור חיי הפניה:**
```
נשלחה ──→ ראיון נקבע ──→ שובצה
                     └──→ לא מתאימה
```

### טבלת `system_settings` — הגדרות מערכת
```sql
key    TEXT PRIMARY KEY  -- 'twilio_account_sid' | 'twilio_auth_token' | ...
value  TEXT
```

## 7. מערכת ההתאמה (AI)

### אלגוריתם ניקוד (lib/matching.ts)

כל זוג מועמדת–משרה מקבל ציון:

| קריטריון | נקודות |
|---------|--------|
| התמחות זהה (מועמדת) | +40 |
| התמחות זהה (משרה) | +50 |
| סטטוס זמינות פעיל | +30 |
| עיר זהה (מועמדת) | +20 |
| עיר זהה (משרה) | +30 |
| יש קורות חיים | +10 |
| סוג תפקיד תואם (סטאג'/בוגרת) | +20 |

### שלב AI — Claude Haiku
לאחר חישוב הניקוד:
1. מסנן מועמדות שכבר הגישו / כבר שובצו
2. לוקח Top 10 לפי ניקוד
3. שולח ל-Claude Haiku לדירוג סמנטי + הסבר בעברית
4. מחזיר Top 3 עם הסבר לכל מועמדת

**Fallback:** אם ה-API כשל — מחזיר Top 3 לפי ניקוד בלבד עם הסבר גנרי.

### תצוגת ממשק ההתאמה
שלושה טאבים בדף `/matching`:
1. **כל ההפניות** — רשימה + עדכון סטטוס + קביעת ראיון
2. **הפעל התאמה** — בחירת משרה → AI מציג מועמדות + יצירת הפניה
3. **יומן ראיונות** — ראיונות מקובצים לפי תאריך (כרטיס לכל יום)

## 9. תהליכים עסקיים

### תהליך רישום מוסד חדש
```
1. מנהל/ת מוסד נרשמת ב-/register/institution
2. נוצר profile עם role='מוסד' + רשומת institutions עם is_approved=false
3. מנהל רשת נכנס ל-/admin/institutions → רואה מוסד "ממתין לאישור"
4. לחיצה על "אשר" → is_approved=true + נשלחת WhatsApp למוסד
5. המוסד יכול כעת לפרסם משרות
```

### תהליך פרסום משרה
```
1. נציג מוסד מאושר → /institution/jobs/new
2. ממלא: כותרת, תיאור, עיר, התמחות, סוג (סטאג'/חלקי/מלא), תאריך התחלה
3. משרה נוצרת עם status='פעילה'
4. מופיעה ב-/jobs לכל המשתמשים
```

### תהליך הגשת מועמדות
```
1. מועמדת גולשת ב-/jobs → מסננת לפי עיר/התמחות
2. לוחצת על משרה → /jobs/[id]
3. ממלאת מכתב כוונות (אופציונלי) → שולחת
4. API בודק כפילות (409 אם כבר הוגשה)
5. נוצרת רשומת application עם status='ממתינה'
6. נשלחת WhatsApp למוסד
7. המועמדת רואה ציר זמן ב-/my-applications
```

### תהליך עדכון סטטוס הגשה
```
1. נציג מוסד נכנס ל-/institution/jobs/[id]
2. רואה הגשות שהתקבלו → לוחץ "צפייה"
3. בוחר סטטוס: נצפתה / התקבלה / נדחתה
4. API מעדכן DB
5. נשלחת WhatsApp למועמדת עם עדכון הסטטוס
6. ציר הזמן של המועמדת מתעדכן
```

### תהליך התאמה (מנהל רשת)
```
1. מנהל רשת נכנס ל-/matching
2. בוחר משרה מהרשימה הנפתחת
3. מערכת AI מנתחת כל המועמדות הרלוונטיות
4. מוצגות Top 3 עם הסבר בעברית
5. מנהל בוחר מועמדת → לוחץ "שלח הפניה"
6. נוצרת רשומת referral עם status='נשלחה'
7. מנהל מעדכן ל-"ראיון נקבע" + תאריך → WhatsApp נשלח למועמדת
8. לאחר הראיון: מעדכן ל-"שובצה" או "לא מתאימה"
```

## 11. אבטחה

### Authentication
- כל ה-API routes מחייבים `supabase.auth.getUser()`
- Middleware מגן על `/admin/*`, `/institution/*`, `/dashboard/*`
- Session cookies מתחדשים אוטומטית

### Authorization
- בדיקת `role` לפני כל פעולה רגישה
- פעולות אדמין: `['מנהל רשת', 'אדמין מערכת']`
- מוסד מאושר בלבד יכול לפרסם משרות
- Ownership verification: מועמדת יכולה לעדכן רק פרופיל שלה

### Input Validation
- Whitelist שדות ב-PATCH (מניעת column injection)
- בדיקת סוג קובץ + גודל ב-CV upload
- Allowed status values בכל endpoint שמעדכן סטטוס

### Environment Variables (לא בקוד)
```
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
ANTHROPIC_API_KEY
TWILIO_ACCOUNT_SID
TWILIO_AUTH_TOKEN
TWILIO_WHATSAPP_FROM
```
